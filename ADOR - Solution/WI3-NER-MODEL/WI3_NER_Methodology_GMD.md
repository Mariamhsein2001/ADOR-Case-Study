# Global Methodology Document (GMD) -  NER Model

## 1. Objective

This document explains how to **fine-tune a general-purpose NER model** to extract financial entities from short, informal chat transcripts.

The base model (`dslim/bert-base-NER`) handles generic named entities (organisations, persons, locations). Fine-tuning adapts it to recognise domain-specific financial labels like `ISIN`, `NOTIONAL`, `UNDERLYING`, `MATURITY`, `BID`, and `PAYMENT_FREQUENCY`.


## 2. Base Model

| Property | Value |
|---|---|
| Model | `dslim/bert-base-NER` (HuggingFace Hub) |
| Architecture | BERT-base (12 layers, 110M parameters) |
| Pre-training | CoNLL-2003 (PER, ORG, LOC, MISC) |
| Input | Tokenised text (WordPiece), max 512 tokens |

### Why BERT?

BERT stands for **Bidirectional Encoder Representations from Transformers**. Unlike older models that read text left-to-right only, BERT reads the entire sentence in **both directions simultaneously**. This gives it a richer understanding of context — for example, the word `"ABC"` means something different in `"BANK ABC"` (a counterparty name) versus `"ABC analysis"` (a generic term). BERT understands this from the surrounding words.

BERT is chosen because it already recognises organisations out of the box, handles short chat messages well within its 512-token limit, runs on CPU without GPU infrastructure, and can be fine-tuned on custom financial labels with relatively little training data.


## 3. Financial Entity Schema

The fine-tuned model must recognise the following custom labels using **BIO tagging**:

| Label | Description | Example |
|---|---|---|
| `COUNTERPARTY` | Name of the trading counterparty | `BANK ABC` |
| `NOTIONAL` | Trade size (numeric + unit) | `200 mio` |
| `ISIN` | International Securities Identification Number | `FR001400QV82` |
| `UNDERLYING` | Instrument name or ticker | `AVMAFC FLOAT 06/30/28` |
| `MATURITY` | Trade tenor or maturity date | `2Y EVG` |
| `BID` | Bid price or spread | `estr+45bps` |
| `OFFER` | Offer price or spread | (absent in this chat = `None`) |
| `PAYMENT_FREQUENCY` | Coupon / interest payment schedule | `Quarterly` |

### What is BIO Tagging?

BIO tagging is a labelling scheme used to mark which words in a sentence belong to an entity and where each entity starts and ends. Every word in the sentence gets exactly one tag:

- **B** = **B**eginning — this word is the **first word** of an entity
- **I** = **I**nside — this word is a **continuation** of the previous entity
- **O** = **O**utside — this word is **not part of any entity**

**Example — applying BIO to our chat message:**

```
Word          Tag
────────────────────────────────
I             O
'll           O
revert        O
regarding     O
BANK          B-COUNTERPARTY   ← first word of the counterparty name
ABC           I-COUNTERPARTY   ← second word, continues the same entity
to            O
do            O
another       O
200           B-NOTIONAL       ← first word of the notional
mio           I-NOTIONAL       ← second word, continues the notional
at            O
2Y            B-MATURITY       ← single-word entity
EVG           I-MATURITY       ← continues maturity
FR001400QV82  B-ISIN           ← single-word entity
estr+45bps    B-BID            ← single-word entity
Quarterly     B-PAYMENT_FREQ   ← single-word entity
```

## 4. Data Preparation

### 4.1 Data Sources
Fine-tuning requires **annotated examples** of financial chat messages. Sources include:
- Internal anonymisedchat logs
- Synthetic data generated by prompting an LLM to produce realistic financial chat messages

### 4.2 Annotation Process

1. Collect raw chat messages (`.txt` format, one message per line)
2. Pre-annotate using the regex rules to bootstrap labels automatically or using Human annotators via an annotation tool — **Doccano**

**Target dataset size**: at least 500 annotated messages per entity class for a robust fine-tune.

### 4.3 Data Format

HuggingFace `datasets` expects tokens and their BIO tags in JSON format:

```json
{
  "tokens":   ["BANK", "ABC", "200", "mio", "FR001400QV82"],
  "ner_tags": ["B-COUNTERPARTY", "I-COUNTERPARTY", "B-NOTIONAL", "I-NOTIONAL", "B-ISIN"]
}
```

## 5. Fine-Tuning Procedure

Fine-tuning follows five sequential steps:

**Step 1 — Load the annotated dataset.** The JSON files produced by the annotation tool are loaded using the HuggingFace `datasets` library. The dataset is split into train, validation, and test sets.

**Step 2 — Train using the HuggingFace Trainer.** The base model's classification head is replaced with a new one sized for 17 labels. Training runs for 5 epochs with a learning rate of 2e-5. After each epoch the model is evaluated on the validation set and the best checkpoint is saved.

**Step 3 — Save the fine-tuned model.** The best checkpoint is saved to disk as a standard HuggingFace model folder and can be loaded directly with `pipeline("ner", model="./ador-ner-financial-final")`.

---

## 6. Evaluation

| Metric | Target |
|---|---|
| Entity-level F1 (macro) | ≥ 0.85 |
| Precision | ≥ 0.88 |
| Recall | ≥ 0.82 |

Evaluation is performed on the held-out test set using **seqeval** (entity-level, not token-level), which penalises partial matches — finding `BANK` but missing `ABC` counts as a failure, not a partial success.
